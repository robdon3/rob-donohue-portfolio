How I Architected This

Financial Worksheet OS — a clean-room, multi-view personal finance ledger system. Inspired by advanced spreadsheet practice (cockpit, debt clocks, runway, paycheck waterfall) but reimplemented as modular Python + a GitHub Pages UI with synthetic data only.

Open live demo Python package

Problem framing

Power users often push Google Sheets past comfort: dual grain (accounts vs holdings), liability-first debt stacks, ATH/ATL time series, expense runway, paycheck waterfalls, and operational policy (“which card today”). That is a system, not a chart.

This demo answers: how do you decompose a worksheet-class finance OS into testable modules with clear contracts—without publishing real personal data?

Constraint: public portfolio artifact. Solution: clean-room synthetic household (“Jordan Lee”) with the same shapes as a mature worksheet, none of the secrets.

Module breakdown

  • ledger/ — Accounts + holdings; bucket rollups; home sleeves.
  • debt/ — Facilities from liability lines; utilization; rough payoff months.
  • snapshots/ — Append-style history; ATH/ATL/YTD/daily derived, not cell overwrites.
  • runway/ — N-month expense fund target, deficit, catch-up rates.
  • cashflow/ — Income cadence → waterfall buckets → scenario multipliers.
  • worksheet/ — Composition root → WorksheetBundle.
  • clients/ — Optional market adapters (still available via --market-demo).

Design trade-offs

DecisionBenefitCost
Synthetic-only public config Safe for GitHub / interviews Not your live personal sheet
Dual grain (accounts + holdings) Matches real worksheet power use More models than “ticker list”
Snapshots as time series ATH/ATL integrity; auditable Must append, not overwrite
Browser reimplements engine Pages-safe zero backend Dual maintenance vs single API
Points/miles as assets Reflects real household wealth forms Valuation is subjective by nature
Simple payoff / runway math Reviewable assumptions Not a lending or advice product

Error resilience & privacy

  • Public path never reads real Drive/Sheets data — config is synthetic YAML/JS seed.
  • Market adapters isolate network failure (unchanged from v1 market demo).
  • Per-module pure functions where possible for unit tests without I/O.
  • CLI fail-fast with structured logs at composition boundaries.

Product maturation path

If this became a product (separate from the portfolio artifact):

  1. Local-first engine — same modules, encrypted SQLite, CSV/Sheets import of structure.
  2. Policy layer — card-of-day and notes as first-class ops, not cells.
  3. Wedge — debt + liquidity + runway for spreadsheet power users (not another pie-chart broker app).
  4. Optional mark-to-market — existing adapters price holdings without owning the ledger core.

Run it

python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
python -m src.main --worksheet
pytest -q

# optional market slice
python -m src.main --market-demo
