How it's built

This is not a generic “dashboard product.” It’s a public, synthetic reconstruction of how I design personal finance tooling: purpose-built areas, custom rollups, automation that matches how I actually work, and a payments sandbox so strategy is tested before it hits the live ledger.

Open live demo Python package

Why it looks like this

I am methodical about this because it has to look good to me and show me what I need — and how to do it. Every sheet / area has a job. Formulas and scripts are custom because the defaults don’t match the decisions I make.

The real workbook is private. This demo uses fictional numbers so the structure and the method are evaluable without publishing personal data.

Journal automation (Apps Script pattern)

In the real Sheets workbook, Apps Script copies the control row into a Journal sheet. The rules are deliberate:

  1. Read today’s snapshot from the control surface.
  2. Scan column A only for dates (other columns may have formulas — ignore them for “last row”).
  3. If today’s date already exists → overwrite that row.
  4. Else → append after the last real date row.

That makes the journal re-runnable. One day, one truth row. The Python module journal/daily.py implements the same contract. Reference copy of the script: automation/copyDataToJournal.gs.js. 

flowchart LR
  CTRL[Control surface totals] --> SCRIPT[copyDataToJournal / journal.upsert]
  SCRIPT --> SCAN[Scan date column only]
  SCAN -->|same day| OW[Overwrite row]
  SCAN -->|new day| AP[Append row]
  OW --> J[(Journal history)]
  AP --> J

Payments testground

Separate from the live debt balances. You set a budget and a rule (avalanche, snowball, directed split). You see remaining balances. Nothing is applied until I decide it is. In the real workbook this is a purpose-built area; here it’s a module with the same idea.

Code map

  • ledger/ — accounts + holdings, rollups
  • journal/ — date-keyed upsert (script twin)
  • debt/ — stack, utilization, payoff clocks
  • payments/ — strategy sandbox
  • snapshots/ — ATH / ATL / YTD from history
  • runway/ — expense fund target / catch-up
  • cashflow/ — paycheck waterfall
  • clients/ — optional market marks (--market-demo)

Trade-offs I accept

  • Custom over generic — slower to build, faster to use daily.
  • Dual grain (accounts + holdings) — more structure, less ambiguity.
  • Synthetic public demo — safe, but not the live sheet.
  • Browser reimplements rules — works on GitHub Pages with no backend.
  • Simple payoff / runway math — clear assumptions, not a bank product.
