How I Architected This

Personal Finance Dashboard — an API-connected Python system with swappable market-data adapters, a clear ETL spine, and pure analytics. Built as original demo code for architecture evaluation.

Open live demo Python package

Problem framing

Hiring narratives often claim “built integrations” without showing boundaries. This demo answers a concrete question: how do you structure a market-data-backed portfolio tool so providers, transforms, and math stay independently testable?

Constraints for a portfolio artifact: no client secrets, no paid market SLA, must run offline for interviews, and still demonstrate production habits (typing, logging, retries, config).

Module breakdown

  • clients/ — Adapter layer. MarketDataClient protocol; mock, yfinance, Alpha Vantage implementations. Network and vendor quirks stop here.
  • models/ — Pydantic contracts (MarketQuote, PriceBar, Holding, PortfolioSnapshot). Stable language between layers.
  • etl/ — Extract via client, transform to DataFrames, join holdings↔quotes, optional load to CSV.
  • analytics/ — Pure functions: valuation, weights, projections, rough vol/Sharpe/drawdown from history.
  • utils/ — Logging setup + exponential backoff decorator (swap for tenacity later without redesign).
  • main.py — CLI composition root. No business logic beyond wiring and presentation.

Design trade-offs

DecisionBenefitCost
Mock as default provider CI, offline interviews, deterministic demos Not “live” until you flip env
Normalize to domain models early Analytics never parse vendor JSON Mapping work per new provider
pandas at ETL boundary Familiar ops for finance-shaped data Heavier than pure lists for tiny demos
Simple compound projections Clear, reviewable assumptions Not Monte Carlo / advice-grade
Browser demo reimplements mock math Pages-safe, zero backend cost Dual maintenance vs single WASM/API host

Error resilience

  • Per-symbol isolation in live clients — one failed ticker does not abort the batch.
  • Retries with backoff on HTTP/transient failures (retry_with_backoff).
  • Factory fail-fast with actionable messages (missing API key → use mock).
  • Structured logs at extract/transform boundaries for operability demos.
  • Degraded path: always available mock; live providers optional.

Scalability notes

This package is a vertical slice, not a distributed system. The architecture still points at scale-out seams:

  • Replace in-process client with async workers + rate-limit queue for bulk symbols.
  • Cache layer (TTL files or Redis) in front of free-tier APIs.
  • Analytics can run as a batch job reading parquet of bars; CLI becomes one consumer of many.
  • Snapshot export already produces a JSON artifact suitable for a thin UI service.

LLM as accelerator

Scaffolding adapters, tests, and portfolio HTML was accelerated with LLM assistance. The system boundaries — adapter interface, pure analytics, config-driven holdings — were architectural decisions first. That split is what makes the demo evaluable and extensible.

python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
python -m src.main --demo
pytest -q

# Live (no key):
FINANCE_PROVIDER=yfinance python -m src.main --demo
